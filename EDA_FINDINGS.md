## 8. Implications for recommender design

- В качестве основного позитивного сигнала разумно использовать "addtocart", а покупки применять как более сильный дополнительный сигнал и для пост-фильтрации уже купленных товаров
- Концентрация популярности умеренная, поэтому одного popularity baseline будет недостаточно
- Свойства товаров выглядят относительно стабильными и подходят для item feature engineering
- Пары user-item с add-to-cart имеют сильную связь с покупкой: conversion cart->transaction = 30.71%. Это подтверждает бизнес-смысл оптимизации add-to-cart
- Для офлайн-оценки лучше делать time-based split, а не случайное перемешивание, так как данные событийные и зависят от времени
- Минимально разумный стек сравнения моделей: popularity baseline, item-to-item co-occurrence или implicit collaborative filtering, затем более сильный ranker для reranking кандидатов

## 9. Recommended offline setup

- Целевой сигнал: `addtocart` как основной позитивный event
- Дополнительный сильный позитивный сигнал: `transaction`
- Исключение из рекомендаций: уже купленные товары пользователя
- Разбиение: строго по времени, train на прошлом, validation на будущем
- Базовые метрики: Recall@K, HitRate@K, MAP@K или NDCG@K
- Бизнес-логика: метрики должны отражать способность модели возвращать товары, которые пользователь добавит в корзину
- Минимальный набор baseline-моделей:
  - popularity
  - co-occurrence item-to-item
  - collaborative filtering или retrieval-кандидаты
  - ranker для reranking кандидатов, если успевает по объему проекта

## 10. Risks and constraints

- Возможна сильная разреженность user-item матрицы
- Возможен cold start по товарам и пользователям
- Свойства товаров могут быть шумными и обновляться во времени
- При случайном split возникнет временная утечка
- Нужно заранее определить, какие события считаются позитивными, а какие использовать только как контекст
