# PROJECT_DECISIONS

## Статус документа

Это предварительный документ с рабочими решениями по проекту.

Он нужен как стартовый ориентир для проектирования репозитория, EDA, моделирования, сервиса и пайплайнов. Это не строгая спецификация. По мере выполнения проекта отдельные решения могут быть пересмотрены после дополнительных экспериментов, проверки качества моделей, анализа ограничений данных и оценки трудозатрат.

Использовать этот документ следует в первую очередь на начальном этапе проекта:
- для выбора разумной стартовой архитектуры
- для определения baseline-подходов
- для согласования offline-оценки
- для построения первой версии репозитория

Если в ходе проекта появятся более обоснованные решения, приоритет должен быть у них, а не у текущей предварительной версии документа.

---

## 1. Контекст проекта

Проект посвящен рекомендательной системе для электронной коммерции.

Бизнес-цель проекта:
- рекомендовать пользователю товары, которые с высокой вероятностью будут добавлены в корзину

Практический смысл:
- оптимизация под add-to-cart ближе к бизнес-пользе, чем оптимизация только под просмотры
- покупки при этом можно использовать как более сильный позитивный сигнал и как основу для пост-фильтрации уже купленных товаров

---

## 2. Наблюдения по данным, влияющие на архитектуру

На основе предварительного EDA были получены следующие важные выводы:

- Основной объем данных составляют просмотры
- События `addtocart` редкие, но содержательно важные
- События `transaction` еще более редкие, но сильно связаны с `addtocart`
- Данные сильно разрежены по пользователям
- У большого числа пользователей очень короткая история взаимодействий
- Популярность товаров распределена умеренно неравномерно, но не настолько, чтобы popularity baseline полностью решал задачу
- Свойства товаров потенциально полезны для item feature engineering
- Не все товары из событийного лога присутствуют в `item_properties`
- Данные имеют временную структуру, поэтому случайный split использовать нельзя

Практический вывод:
- проект нужно строить вокруг time-based evaluation
- система должна иметь fallback-стратегии
- модель должна учитывать разреженность и cold start
- baseline-модели нужно делать простыми, интерпретируемыми и воспроизводимыми

## 3. Предварительная постановка задачи

### Целевой сигнал

Основной позитивный сигнал:
- `addtocart`

Дополнительный сильный позитивный сигнал:
- `transaction`

Контекстный сигнал:
- `view`

Предварительная логика использования событий:
- `view` использовать для формирования кандидатов, анализа интереса, co-occurrence и пользовательской истории
- `addtocart` использовать как основной таргет для offline-оптимизации
- `transaction` использовать как усиленный позитивный сигнал и для исключения уже купленных товаров из будущих рекомендаций

### Что именно предсказываем

Предварительная задача:
- по истории пользователя предсказать top-K товаров, которые он с наибольшей вероятностью добавит в корзину в будущем временном окне

---

## 4. Предварительная стратегия моделирования

На старте проекта разумно опираться не на одну сложную модель, а на последовательное сравнение нескольких подходов.

### Этап 1. Простые baseline-модели

Обязательные baseline:
- global popularity
- personalized popularity по истории пользователя
- item-to-item co-occurrence

Почему это нужно:
- baseline должны задавать нижнюю границу качества
- они дешевы в реализации
- их легко интерпретировать
- они хорошо подходят для разреженных ecommerce-данных

### Этап 2. Более сильный retrieval

Кандидаты на роль retrieval-модели:
- item-to-item co-occurrence с весами по типам событий
- implicit collaborative filtering
- matrix factorization для неявной обратной связи

Предварительное решение:
- стартовать с item-to-item co-occurrence как с наиболее надежного и прозрачного подхода
- затем, если позволяет время, добавить collaborative filtering и сравнить

### Этап 3. Reranking

Если проектный объем позволит, после retrieval добавить reranking-контур:
- для каждого пользователя генерировать ограниченный пул кандидатов
- затем ранжировать их более сильной моделью

Возможные признаки для reranking:
- число просмотров user-item
- факт add-to-cart в прошлом
- давность последнего взаимодействия
- популярность товара
- статистики по товару
- признаки из `item_properties`
- признаки категории
- простые агрегаты по пользователю и товару

Предварительное решение:
- reranking считать желательным, но не обязательным первым шагом
- если сроки ограничены, сделать качественный retrieval и хороший production pipeline важнее, чем перегружать проект сложным ranker

## 5. Предварительная offline-оценка

### Разбиение данных

Использовать только time-based split.

Предварительная схема:
- train на более раннем временном интервале
- validation на более позднем временном интервале
- при необходимости test на самом позднем интервале

Почему это важно:
- данные событийные
- поведение пользователя зависит от времени
- случайный split приведет к утечке и завышенной оценке качества

### Целевая логика оценки

Оценивать способность модели возвращать товары, которые пользователь реально добавит в корзину в будущем окне.

### Предварительный набор метрик

Основные:
- Recall@K
- HitRate@K

Дополнительные:
- MAP@K
- NDCG@K

Предварительное решение:
- основной метрикой сделать Recall@K
- дополнительно логировать HitRate@K и одну ranking-метрику, например NDCG@K

### Значения K

Предварительно рассмотреть:
- K = 5
- K = 10
- K = 20

Окончательное значение выбрать после первых экспериментов и с учетом бизнес-логики сервиса.

## 6. Предварительная логика формирования обучающих данных

### Пользовательская история

Использовать только события, произошедшие до момента предсказания.

### Позитивные примеры

Основной позитив:
- user-item пары с `addtocart`

Дополнительный усиленный позитив:
- user-item пары с `transaction`

### Негативные примеры

Предварительно:
- явных негативов в данных нет
- негативы формировать из непросигналенных кандидатов в рамках окна наблюдения
- при необходимости использовать negative sampling

### Исключения из рекомендаций

На инференсе желательно исключать:
- уже купленные товары пользователя

На этапе экспериментов это решение может быть уточнено.

## 7. Использование item features и category information

### Item properties

Предварительное решение:
- использовать `item_properties` как источник item features
- не пытаться сразу использовать все 1104 свойства без отбора
- сначала выделить наиболее частые и интерпретируемые свойства
- отдельно обработать `categoryid` и `available`

Важно:
- часть товаров из событий отсутствует в `item_properties`
- значит система не должна зависеть от item features как от обязательного источника

### Category tree

Предварительное решение:
- использовать категории как вспомогательные признаки и для анализа
- не строить на первом этапе слишком сложную иерархическую модель по дереву категорий
- если будет полезно, добавить category-level fallback или category-aware popularity

---

## 8. Cold start и fallback-логика

### Cold start пользователя

Для новых или почти новых пользователей:
- global popularity
- recent popularity
- category-aware popularity, если получится надежно извлечь категорию контекста

### Cold start товара

Для новых или малоисторичных товаров:
- опора на item features
- category-based heuristics
- fallback через более общие популярные товары

### Недостаток истории

Если у пользователя слишком мало событий:
- использовать упрощенную персонализацию
- при необходимости переключаться на fallback

Предварительное решение:
- fallback-сценарии считать обязательной частью проекта
- это важно и для качества, и для production API

---

## 9. Предварительная структура экспериментов и MLflow

MLflow должен использоваться не формально, а как центральный инструмент управления экспериментами.

Что логировать:
- название эксперимента
- тип модели
- временной split
- параметры подготовки данных
- параметры модели
- метрики Recall@K, HitRate@K, NDCG@K или MAP@K
- путь к артефактам
- сериализованную модель
- при необходимости примеры рекомендаций

Предварительное решение:
- сравнение моделей организовать через единый эксперимент
- имена запусков делать осмысленными и воспроизводимыми
- сохранить лучшую модель как отдельный артефакт и использовать ее в сервисе

---

## 10. Предварительная структура production-решения

### API

Минимально разумный API:
- endpoint healthcheck
- endpoint для получения рекомендаций по `visitorid`
- при необходимости endpoint с метаданными версии модели

Пример логики inference:
1. получить пользователя
2. загрузить историю пользователя
3. если история достаточна, построить персональные рекомендации
4. если история недостаточна, отдать fallback-рекомендации
5. исключить уже купленные товары, если это предусмотрено финальной логикой
6. вернуть top-K рекомендаций

### Docker

Сервис должен стабильно подниматься в Docker.

Предварительное решение:
- сделать простой и надежный Dockerfile
- не усложнять архитектуру лишними сервисами, если они не нужны для демонстрации проекта

---

## 11. Предварительная логика Airflow DAG

DAG должен быть не формальным, а отражать понятный цикл переобучения.

Минимальные шаги:
- загрузка исторических данных
- preprocessing
- построение train/validation выборок
- обучение модели
- оценка модели
- логирование в MLflow
- сохранение артефактов
- обновление актуальной версии модели

Предварительное решение:
- DAG делать линейным и понятным
- важнее воспроизводимость и читаемость, чем избыточная оркестрационная сложность

---

## 12. Предварительная стратегия мониторинга

Нужно мониторить не только сервис, но и качество его работы как рекомендательной системы.

### Технические метрики сервиса

- latency
- request count
- error rate
- uptime
- timeout count

### Продуктовые и модельные метрики

- доля fallback-рекомендаций
- доля пустых рекомендаций
- число уникальных рекомендованных товаров
- распределение длины списков рекомендаций
- версия модели в метриках или логах

Предварительное решение:
- в коде сервиса сразу заложить экспорт технических метрик
- отдельно в документации описать, какие продуктовые метрики нужно отслеживать в реальной эксплуатации

## 13. Предварительная структура репозитория

Репозиторий должен быть организован так, чтобы ревьюеру было легко:
- прочитать основной notebook
- понять ход экспериментов
- запустить сервис
- поднять MLflow
- запустить Airflow DAG
- воспроизвести окружение

Предварительная логика структуры:
- `notebooks/` для EDA и экспериментов
- `src/` для reusable-кода
- `services/` или `app/` для API
- `airflow/` для DAG
- `scripts/` для запуска и утилит
- `models/` для артефактов локальной сборки, если нужно
- `README.md` как центральная документация

Точная структура может быть уточнена по мере проектирования.

## 15. Что считать хорошим улучшением, но не жестким требованием

Если хватит времени, полезно добавить:
- более сильный retrieval через collaborative filtering
- reranking кандидатов
- более аккуратную работу с item features
- category-aware fallback
- анализ ошибок моделей
- сравнение качества по разным типам пользователей
- раздельный анализ warm users и cold users

## 16. Открытые вопросы, которые могут уточниться по ходу проекта

Эти решения пока не зафиксированы окончательно:
- какая именно модель retrieval окажется лучшей
- нужен ли полноценный reranking в финальной версии
- какие признаки из `item_properties` реально улучшают качество
- какие значения K стоит выбрать как основные
- нужно ли исключать только купленные товары или также уже просмотренные
- как именно лучше формировать negative samples
- нужен ли отдельный test split помимо validation

---

## 17. Итоговая предварительная позиция

На текущем этапе наиболее разумная стартовая стратегия выглядит так:

- оптимизировать рекомендации под `addtocart`
- делать оценку строго по времени
- начать с сильных и простых baseline-моделей
- затем добавить retrieval сильнее popularity
- при наличии времени добавить reranking
- строить production-часть вокруг простого, надежного и воспроизводимого API
- сделать Airflow и MLflow не декоративными, а реально встроенными в проект
- учитывать cold start и fallback с самого начала

Этот документ должен использоваться как стартовая проектная рамка, но не как окончательная спецификация.
